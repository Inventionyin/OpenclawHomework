# 飞书桥梁服务使用说明

## 1. 作用

桥梁服务负责把飞书群里的指令转换成 GitHub Actions 调度请求。

当前链路：

```text
飞书消息
  -> POST /webhook/feishu
  -> 解析 /run-ui-test main contracts
  -> 可选：调用 OpenClaw 解析自然语言
  -> 调用 GitHub workflow_dispatch
  -> 触发 UI Automation Dispatch
  -> GitHub Actions 拉取电商前端和 UItest
  -> GitHub Actions 启动电商前端并生成测试报告
  -> 可选：轮询 GitHub Actions 结果并主动发回飞书
```

## 2. 当前已实现能力

- 支持飞书 challenge 校验响应
- 支持解析飞书事件中的 `event.message.content`
- 支持简单指令：

```text
/run-ui-test main contracts
```

- 支持三个运行模式：
  - `contracts`
  - `smoke`
  - `all`
- 支持用户 allowlist 授权
- 支持调用 GitHub API 触发 Actions
- 支持可选 OpenClaw 自然语言解析
- 支持查询 GitHub Actions run 链接
- 支持可选飞书主动结果通知
- 支持飞书 interactive card 结果卡片
- 支持普通聊天：`你好`、`帮助`、自由问答
- 支持 `绑定我`，把测试触发权限限制到当前飞书用户
- 支持 Agent Router，把消息分流到聊天、UI 测试、运维、文档、记忆等轻量 agent
- 支持安全记忆文件和技能文档，帮助新 AI 接手项目
- 支持 `/status`、`/health`、`/watchdog`、`/logs` 等白名单运维查询
- 支持文字生图和图片编辑：可以直接生图，也可以发旧照片让机器人修复

自然语言示例：

```text
帮我跑一下 main 分支的 UI 自动化冒烟测试
```

启用 OpenClaw 解析后，会转换成：

```json
{"targetRef":"main","runMode":"smoke"}
```

## 3. 本地启动方式

在项目根目录执行：

```powershell
$env:GITHUB_TOKEN = (Get-Content -Raw "C:\Users\殷文炫\Desktop\Github_TOKEN.txt").Trim()
npm run bridge:feishu
```

如果要在本地启用 OpenClaw 自然语言解析：

```powershell
$env:OPENCLAW_PARSE_ENABLED = "true"
$env:OPENCLAW_MODEL = "xfyun/astron-code-latest"
npm run bridge:feishu
```

默认监听：

```text
http://127.0.0.1:8787
```

健康检查：

```text
GET http://127.0.0.1:8787/health
```

飞书 Webhook 地址：

```text
POST http://127.0.0.1:8787/webhook/feishu
```

如果要让飞书公网访问本地服务，需要再使用内网穿透工具，把本地 `8787` 暴露成公网 HTTPS 地址。

当前已经部署到云服务器：

```text
https://openclaw.evanshine.me/health
https://openclaw.evanshine.me/webhook/feishu
https://openclaw.evanshine.me/webhook/feishu/openclaw
https://openclaw.evanshine.me/webhook/feishu/hermes
https://hermes.evanshine.me/webhook/feishu
```

服务器上实际 Node 服务端口是 `8788`，因为 `8787` 已被原有 Python 服务占用。外部访问不需要带端口，统一走 Nginx 的 HTTPS `443` 端口。

路由说明：

- `/webhook/feishu`：兼容旧 OpenClaw 机器人，默认 OpenClaw。
- `/webhook/feishu/openclaw`：OpenClaw 专用机器人入口，默认 OpenClaw，Hermes 作为 fallback。
- `/webhook/feishu/hermes`：Hermes 专用机器人入口，默认 Hermes，OpenClaw 作为 fallback。
- `https://hermes.evanshine.me/webhook/feishu`：Hermes 独立域名入口，等价于上面的 Hermes 专用入口，更推荐填到 Hermes 机器人后台。

## 4. 本地模拟飞书请求

不接飞书时，可以用 PowerShell 模拟：

```powershell
$body = @{
  event = @{
    sender = @{
      sender_id = @{
        open_id = "local-user"
      }
    }
    message = @{
      content = '{"text":"/run-ui-test main contracts"}'
    }
  }
} | ConvertTo-Json -Depth 10

Invoke-RestMethod `
  -Method Post `
  -Uri "http://127.0.0.1:8787/webhook/feishu" `
  -ContentType "application/json" `
  -Body $body
```

成功时会返回：

```json
{
  "ok": true,
  "message": "UI 自动化测试已触发：分支 main，模式 contracts",
  "actionsUrl": "https://github.com/Inventionyin/OpenclawHomework/actions/workflows/ui-tests.yml"
}
```

```powershell
$body = @{
```

成功时返回里的 `commandSource` 会是：

```text

```

## 5. 授权配置

如果不配置 `FEISHU_ALLOWED_USER_IDS`，本地开发默认允许所有用户触发。

如果要限制用户：

```powershell
$env:FEISHU_ALLOWED_USER_IDS = "open_id_1,open_id_2"
```

只有 allowlist 里的飞书用户 ID 才能触发 UI 自动化。

如果还不知道自己的飞书 `open_id`，推荐先打开：

```powershell
$env:FEISHU_REQUIRE_BINDING = "true"
```

这样在没有绑定用户前，任何测试触发请求都会被拒绝；你在飞书里发送 `绑定我` 后，服务会把当前发送人的 `open_id` 写入 allowlist。

## 6. 飞书应用配置时要填什么

飞书开放平台里需要配置：

- 机器人能力
- 事件订阅
- 接收消息事件
- 请求地址：公网 HTTPS 的 `/webhook/feishu`

示例：

```text
https://your-public-domain.example.com/webhook/feishu
```

## 7. Agent Router、记忆和技能

桥梁服务支持轻量 Agent Router。它不是多个并发 OpenClaw CLI，而是在同一个 Node 服务里按意图分工，避免再次出现多个模型请求抢同一个 session 的问题。

当前 agent 分工：

```text
chat-agent    普通聊天，不默认携带完整项目记忆
ui-test-agent UI 自动化触发，继续走原有 GitHub Actions 流程
ops-agent     安全状态检查，只允许白名单动作
doc-agent     回答任务进度、交接和文档问题
memory-agent  读取或记录非敏感记忆
```

可用命令：

```text
/status
/health
/watchdog
/logs
/peer-status
/peer-health
/peer-logs
/peer-restart
/peer-repair
/memory
/memory search session lock
/memory remember OpenClaw 已加串行队列
老师任务还差哪些
帮我跑一下 main 分支的 UI 自动化冒烟测试
```

也支持更像聊天的自然语言说法：

```text
你好
帮助
你会做什么
怎么玩
你现在内存多少
你硬盘还剩多少
看看哪些东西占硬盘
khoj 可以清理吗
确认清理第 1 个
你现在卡不卡
看看 Hermes 的服务器状态
OpenClaw 硬盘还剩多少
重启你自己
修复你自己
重启 Hermes
修复 OpenClaw
```

危险操作有置信度保护：明确的“重启你自己”“修复 Hermes”会执行；像“你重起一下”“帮我搞一下”这类模糊说法会先要求确认，不会直接重启或修复。

硬盘清理是两步流程：先说“看看哪些东西占硬盘”或“khoj 可以清理吗”，机器人只做盘点并保存本轮候选项；再说“确认清理第 1 个”或“清理 khoj”，才会执行白名单内的清理动作。当前白名单包括 Khoj 目录、npm 缓存、旧日志和临时文件，不支持从自然语言执行任意删除命令。

飞书回复展示增强：

```text
FEISHU_REPLY_FOOTER_ELAPSED=true
FEISHU_REPLY_FOOTER_STATUS=true
FEISHU_TIMING_LOG_ENABLED=true
```

开启后，桥接服务会在文本回复底部追加状态和耗时，并在服务日志里记录收到消息、路由、模型调用、发送飞书等阶段耗时。OpenClaw 官方 Feishu channel 也可以同时开启：

```bash
openclaw config set channels.feishu.streaming true
openclaw config set channels.feishu.footer.elapsed true
openclaw config set channels.feishu.footer.status true
```

普通聊天真流式回复：

```text
FEISHU_CHAT_STREAMING_ENABLED=true
STREAMING_MODEL_BASE_URL=https://maas-coding-api.cn-huabei-1.xf-yun.com/v2
STREAMING_MODEL_API_KEY=...
STREAMING_MODEL_ID=astron-code-latest
STREAMING_MODEL_ENDPOINT_MODE=auto
FEISHU_STREAM_UPDATE_INTERVAL_MS=800
```

说明：

- 只对普通聊天启用流式；`/status`、UI 自动化、修复服务器等命令仍走稳定的确定性回复。
- 桥梁会优先尝试 `/responses`，如果接口不是 SSE 或不兼容，会自动降级到 `/chat/completions`。
- 讯飞 CodingPlan 当前实测可用的是 `/chat/completions` 的 `text/event-stream`，`/responses` 会返回 One API HTML 页面，不适合作为流式接口。
- 飞书侧先发一条“正在思考...”占位消息，再用 `PATCH /im/v1/messages/{message_id}` 更新同一条消息，避免刷屏。
- 流式失败时会自动回退到原来的 OpenClaw/Hermes CLI 一次性回复。

图片生成和图片编辑：

```text
生成一张图片：赛博风电商客服机器人海报，蓝白科技风
/image 极简科技风商品主图，白底，高级电商摄影
```

旧照片修复或改图支持两种用法：

```text
发图片时一起写：修复这张旧照片，增强清晰度，保留原貌
先发一张图片，再发：修复刚才那张旧照片
```

图片相关环境变量：

```text
IMAGE_MODEL_BASE_URL=https://img2.suneora.com
IMAGE_MODEL_API_KEY=...
IMAGE_MODEL_ID=auto
IMAGE_MODEL_SIZE=1024x1024
FEISHU_IMAGE_MEMORY_TTL_MS=600000
```

说明：

- 文字生图走 `/v1/images/generations`。
- 带图修复和改图走 `/v1/images/edits`，需要图片中转站支持图片编辑接口。
- 机器人会先发“开始生成图片”或“开始处理图片”，完成后用飞书图片卡片返回结果。
- “先发图再说修复刚才那张”只记住当前聊天和当前发送者最近 10 分钟内的图片，不会跨人误用。

记忆文件位置：

```text
data/memory/
```

技能说明位置：

```text
docs/skills/
```

玩法和二改路线：

```text
docs/Agent玩法与二改优化路线.md
```

安全限制：

- 测试、运维、文档、记忆命令都需要授权用户。
- 普通聊天不会默认携带完整 `data/memory/` 内容，避免未授权用户套出项目状态。
- 记忆系统拒绝保存疑似密钥、Token、密码。
- 运维 agent 只允许 `status`、`health`、`watchdog`、`logs` 和 `peer-*` 白名单动作，不支持飞书任意 shell。
- `peer-*` 是双服务器互修入口：Hermes 可以检查或修复 OpenClaw，OpenClaw 可以检查或修复 Hermes。远端 SSH 被 `authorized_keys command=...` 限制为 `scripts/peer-control.js`，只能执行状态、日志、重启、拉代码测试重启这几类动作。
- 群聊里没有 @ 机器人时，默认忽略普通聊天、文档、记忆问题；只允许明确的测试触发或显式运维命令通过。
- `不要 /run-ui-test main smoke`、`请问 /run-ui-test main smoke 怎么用` 这类否定或教程语境不会触发 GitHub Actions。

## 8. 当前限制

如果要让服务主动把最终结果发回飞书，需要配置飞书应用凭证：

```text
FEISHU_RESULT_NOTIFY_ENABLED=true
FEISHU_APP_ID=...
FEISHU_APP_SECRET=...
```

可选配置：

```text
FEISHU_NOTIFY_RECEIVE_ID_TYPE=chat_id
FEISHU_NOTIFY_RECEIVE_ID=...
FEISHU_AUTOMATION_RECEIPT_ENABLED=false
FEISHU_DEDUP_ENABLED=true
FEISHU_DEDUP_TTL_MS=300000
GITHUB_RUN_NOTIFY_ATTEMPTS=60
GITHUB_RUN_NOTIFY_INTERVAL_MS=10000
```

不配置 `FEISHU_NOTIFY_RECEIVE_ID` 时，服务会优先使用飞书事件里的 `chat_id`，没有 `chat_id` 时再使用发送人的 `open_id`。
如果希望飞书里更像 ChatGPT 一样“一问一答”，可以设置 `FEISHU_AUTOMATION_RECEIPT_ENABLED=false`，关闭“已收到，正在后台触发”的即时提示，只保留最终结果卡片。
`FEISHU_DEDUP_ENABLED=true` 会忽略短时间内重复投递的同一条飞书事件，避免同一条消息触发两次回复或两次测试。

测试完成后也可以同步发邮件。邮件通知由桥梁服务在 GitHub Actions 完成后发送，和飞书报告共用同一次完成状态轮询。邮件发送失败只记录日志，不影响飞书报告。

```text
EMAIL_NOTIFY_ENABLED=true
SMTP_HOST=smtp.qq.com
SMTP_PORT=465
SMTP_SECURE=true
SMTP_USER=your-email@qq.com
SMTP_PASS=邮箱 SMTP 授权码
EMAIL_FROM=your-email@qq.com
EMAIL_TO=receiver@example.com,another@example.com
```

注意：`SMTP_PASS` 通常不是邮箱登录密码，而是邮箱服务商生成的 SMTP 授权码。不要把这些值写入 GitHub 仓库，只放在服务器的 `/etc/openclaw-feishu-bridge.env` 或 `/etc/hermes-feishu-bridge.env`。

从 2026-05-04 开始，邮件通知支持“邮箱动作注册表”，不再只依赖单一 `EMAIL_TO`。主配置文件：

```text
config/mailbox-action-map.json
```

第一阶段已启用的真实动作绑定：

```text
report -> watchee.report@claw.163.com
replay -> evasan.replay@claw.163.com
files  -> agent3.files@claw.163.com
daily  -> agent4.daily@claw.163.com
```

第二阶段已启用的业务动作绑定：

```text
account -> evasan.account@claw.163.com
shop    -> evasan.shop@claw.163.com
support -> agent4.support@claw.163.com
```

说明：

```text
report -> UI 自动化结果汇总
replay -> 失败回放入口、截图、trace 提醒
files  -> Allure / Playwright artifact 入口通知
daily  -> 每日自动化测试汇总
account -> 注册、登录、验证码、账号体系专项测试
shop    -> 商品、购物车、下单、订单流专项测试
support -> AI 客服、售后、咨询流专项测试
```

第二阶段不是自动猜业务类型，而是走显式动作提示。推荐这样用：

```text
1. GitHub Actions workflow_dispatch 里传 mailbox_action
2. 本地脚本触发时带 --mailbox-action
3. 代码里直接给 job.mailboxAction 或 config.inputs.mailbox_action
```

示例：

```powershell
node scripts/trigger-ui-tests.js `
  --token ghp_example `
  --run-mode smoke `
  --target-repository Inventionyin/UItest `
  --app-repository dengzhekun/projectku-web `
  --mailbox-action support
```

这样一次成功或失败通知，除了默认的 `report / replay / files` 之外，还会额外发到对应业务邮箱。

如果只想临时覆盖某个动作的收件邮箱，可以使用环境变量：

```text
MAILBOX_ACTION_REPORT_TO=...
MAILBOX_ACTION_REPLAY_TO=...
MAILBOX_ACTION_FILES_TO=...
MAILBOX_ACTION_DAILY_TO=...
MAILBOX_ACTION_ACCOUNT_TO=...
MAILBOX_ACTION_SHOP_TO=...
MAILBOX_ACTION_SUPPORT_TO=...
```

改完后重启桥梁服务即可生效。

如果使用 Hermes 独立飞书机器人，还需要在服务器环境变量里配置：

```text
HERMES_FEISHU_APP_ID=...
HERMES_FEISHU_APP_SECRET=...
```

这样 `https://hermes.evanshine.me/webhook/feishu` 收到的消息会用 Hermes 机器人的身份回复，而不是 OpenClaw 机器人的身份。

## 9. Watchdog 守护巡检

仓库提供了 `scripts/server-watchdog.js` 和 `scripts/install-watchdog.sh`。

用途：

- 定时检查桥梁服务 `/health`。
- 健康检查失败时自动 `systemctl restart` 对应服务。
- 扫描 Nginx access log 中的飞书 webhook POST。
- 发现非 `200` 回调或请求风暴时记录原因，避免再次出现夜间消息轰炸时没人发现。

本地手动运行：

```powershell
npm run watchdog -- --service openclaw-feishu-bridge --health-url http://127.0.0.1:8788/health
```

服务器安装定时器：

```bash
bash scripts/install-watchdog.sh \
  --unit-name openclaw-homework-watchdog \
  --bridge-service openclaw-feishu-bridge \
  --env-file /etc/openclaw-feishu-bridge.env
```

Hermes 服务器把 `unit-name`、`bridge-service`、`env-file` 换成 Hermes 对应值。

## 10. 结果卡片和报告入口

当前结果通知默认使用飞书卡片形式，包含：

- GitHub Actions 按钮
- Allure 报告按钮
- 分支、模式、结论
- 成功/失败颜色区分

Allure 报告按钮会跳转到本次 GitHub Actions run 的 Artifacts 区域。进入后下载 `allure-report` 或相关 artifact，就能查看更完整的测试报告、截图和日志。

