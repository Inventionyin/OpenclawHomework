# 飞书触发 GitHub UI 自动化流程图实现拆解

## 1. 这张流程图在讲什么

这张图描述的是一个完整的远程 UI 自动化调度系统：

```text
测试/开发人员
  -> 在飞书群发送指令
  -> 飞书服务器推送 Webhook
  -> 桥梁服务解析并鉴权
  -> 调用 GitHub API
  -> GitHub Actions 执行 UI 自动化
  -> 收集报告
  -> 把结果发回飞书群
```

简单说，它不是让人在 GitHub 页面手动点按钮，而是让飞书消息变成一次自动化测试执行。

## 2. 核心模块

| 模块 | 作用 | 当前状态 |
|---|---|---|
| 飞书群机器人 | 接收用户指令，例如 `/run-ui-test main` | 未完成 |
| 飞书服务器 | 把群消息以 Webhook 方式推送出来 | 未完成 |
| 桥梁服务 | 接收 Webhook，解析命令，做权限校验 | 已完成本地版本 |
| GitHub API | 触发 `workflow_dispatch` 或 `repository_dispatch` | 已完成 `workflow_dispatch` |
| GitHub Actions | 拉代码、装依赖、跑 UI 测试、生成报告 | 已完成 contracts + Allure |
| 结果回传 | 将成功/失败和报告链接发回飞书 | 未完成 |

## 3. 推荐先做哪条路线

流程图里有两个关键触发方式：

- `workflow_dispatch`
- `repository_dispatch`

建议第一版先用 `workflow_dispatch`。

原因：

- 配置最简单
- GitHub Actions 页面可以手动测试
- GitHub API 调用直接
- 适合课程作业和演示

等第一版跑通后，再考虑 `repository_dispatch`。

## 4. 最小可运行版本

第一版不要一次做完整飞书闭环，可以拆成三步：

### 第一步：GitHub Actions 能手动运行

目标：

```text
在 GitHub 页面点击 Run workflow，能跑 UI 自动化测试。
```

要完成：

- [ ] 创建 Playwright 测试项目
- [ ] 创建 `.github/workflows/ui-tests.yml`
- [ ] 支持 `workflow_dispatch`
- [ ] 在 GitHub Actions 页面手动运行成功

这一阶段先不用飞书，也不用桥梁服务。

### 第二步：脚本能触发 GitHub Actions

目标：

```text
在本地运行脚本，能触发 GitHub Actions。
```

要完成：

- [ ] 创建 GitHub Token
- [ ] 编写 `scripts/trigger-ui-tests.js`
- [ ] 调用 GitHub REST API
- [ ] GitHub Actions 页面出现新的执行记录

这一阶段先不用飞书，但已经证明“外部程序可以触发 GitHub Actions”。

### 第三步：桥梁服务接收指令并触发脚本

目标：

```text
桥梁服务收到 HTTP 请求后，触发 GitHub Actions。
```

要完成：

- [ ] 创建 Flask 或 FastAPI 服务
- [ ] 提供 `/webhook/feishu` 接口
- [ ] 解析指令，例如 `/run-ui-test main`
- [ ] 校验用户或签名
- [ ] 调用 GitHub API
- [ ] 返回启动成功或失败消息

这一阶段跑通后，就可以接飞书。

## 5. 流程图对应任务清单

### 任务 1：用户在飞书群发指令

示例：

```text
/run-ui-test main
```

需要解析出：

- 命令：`run-ui-test`
- 分支：`main`

验收标准：

- 能从一段文本里识别出要跑 UI 测试
- 能识别测试分支

### 任务 2：飞书服务器推送 Webhook

飞书收到群消息后，会把事件推送给你的服务。

你的服务需要有一个公网可访问地址，例如：

```text
https://your-domain.com/webhook/feishu
```

本地开发时可以使用内网穿透工具。

验收标准：

- 飞书发消息后，你的服务能收到 HTTP 请求

### 任务 3：桥梁服务解析指令

桥梁服务可以用 Flask 或 FastAPI。

职责：

- 接收飞书 Webhook
- 提取消息内容
- 判断是不是 `/run-ui-test`
- 判断分支是否合法
- 判断发送人是否有权限

验收标准：

- 合法指令继续执行
- 非法指令返回错误提示
- 未授权用户不能触发测试

### 任务 4：调用 GitHub API

使用 GitHub API 触发 workflow：

```text
POST /repos/Inventionyin/OpenclawHomework/actions/workflows/ui-tests.yml/dispatches
```

请求体示例：

```json
{
  "ref": "main",
  "inputs": {
    "test_env": "staging"
  }
}
```

验收标准：

- API 返回成功
- GitHub Actions 页面出现新任务

### 任务 5：GitHub Actions 执行流水线

流水线步骤：

1. 拉取指定分支代码
2. 配置测试环境
3. 安装依赖和浏览器
4. 执行 UI 自动化脚本
5. 生成测试报告

验收标准：

- 成功时 workflow 绿色通过
- 失败时能看到失败日志
- 有报告 artifact 可下载

### 任务 6：桥梁服务收集测试结果

第一版可以只返回 GitHub Actions 链接。

增强版可以查询：

- workflow run 状态
- 成功或失败
- 报告 artifact
- 失败日志

验收标准：

- 能知道测试是否启动成功
- 能把 Actions 链接发回飞书

### 任务 7：结果发回飞书群

成功消息示例：

```text
UI 自动化测试已启动
分支：main
状态：运行中
查看结果：https://github.com/Inventionyin/OpenclawHomework/actions
```

失败消息示例：

```text
UI 自动化测试触发失败
原因：GitHub Token 权限不足
```

验收标准：

- 飞书群能收到启动结果
- 测试失败时能看到失败原因或日志链接
