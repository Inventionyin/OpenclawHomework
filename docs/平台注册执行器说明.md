# 平台注册执行器说明

## 1. 目标与定位

平台注册执行器采用统一方案：

- 通用框架（负责任务编排、步骤执行、日志归档、结果输出）
- 平台适配器（负责各平台注册页的字段映射、流程差异与校验规则）

通过这套结构，可以在不改动主流程的前提下，按平台逐步扩展自动化注册测试能力。

## 2. 安全与边界

默认策略如下：

- 默认 `dry-run`：仅做流程演练与可达性验证，不提交真实注册。
- 仅允许自有/测试/沙箱平台：禁止把执行器用于未知或外部真实业务平台的批量注册。
- 严禁绕过风控：不做验证码绕过、不做限制规避、不做异常流量模拟。

本方案服务于测试工程、回归验证与教学/作业场景，不用于任何违规用途。

## 3. 架构摘要

建议执行链路：

1. 任务入口接收“平台注册测试”请求。
2. 通用框架加载目标平台适配器。
3. 先跑 `dry-run` 预检（页面可达、字段定位、验证码收件链路、结果归档链路）。
4. 仅在明确授权的测试环境下，按策略切换到真实提交模式。
5. 产出统一报告（成功率、失败点、截图/trace、邮箱归档位置）。

## 4. 首个适配器示例：`projectku-web`

`projectku-web` 作为第一批平台适配器示例，建议覆盖：

- 注册入口定位与页面健康检查
- 邮箱字段、密码字段、确认字段映射
- 验证码邮件收取与有效期校验
- 已注册邮箱、错误验证码、重复发送等关键负例
- 统一产物归档到既有 QA/邮箱资产链路

该适配器用于验证“通用框架 + 适配器”模式是否可复用，后续再按同样接口扩展更多测试站点。

## 5. 与现有资产的关系

平台注册执行器不是独立烟囱，需复用已有文档中沉淀的资产能力：

- QA 数据资产（注册/验证码相关用例可直接复用与扩写）
- 邮箱动作链路（`verify`、`account` 等动作可承接验证码与账号测试）
- 报告归档链路（报告、附件、复盘进入既有归档邮箱与数据目录）

这样可以把“平台注册测试”纳入统一的自动化与评测体系，避免重复建设。
